# delivery-message — 数据组交付发布消息模板

把"做完的东西"包装成对外发布的标准消息。统一格式 = 申请人一眼看懂谁交付了什么；也方便群里搜索归档。

---

## 核心契约（不要违反）

1. **机构称呼固定为「GL.iNet 数据团队」**——不要简写为"我们组"、"DT 同学"、"数据组"
2. **「核心结论：」字段名固定**——不要换成"结论"、"重点"、"摘要"，否则破坏自动归档
3. **必须 @ 到申请人**——飞书 / 企业微信里使用真实 mention，不要写成纯文字 `@张三`
4. **结尾必须有"出问题来找我们"那句**——这是 SLA 信号，不能省

---

## 工作流（用户触发后按序执行）

### Step 1：收齐 5 个槽位

执行模板填空之前必须确认这 5 个变量。用户没给就**逐项问**（一次问完，不要一来一回）：

| 槽位 | 含义 | 示例 |
|---|---|---|
| `ticket_id` | 需求序号（与工单系统一致） | `DT-2026-0427-01` / `#1284` |
| `ticket_title` | 需求标题（与工单完全一致，别擅自改写） | `日本 Wi-Fi 6 路由器竞品分析` |
| `doc_type` | 产物类型（见下面对照表） | `市场调研报告` |
| `requester` | 申请人（IM 里能 @ 到的人） | `@张三` |
| `core_findings` | 核心结论 1–8 条 bullet | 见下面"核心结论书写约束" |

### Step 2：让用户从 7 种官方 doc_type 中选一个

数据组对外交付**只有这 7 种产物类型**。用户没指定时，列出这份编号菜单让对方选号；选完之后`doc_type`就用菜单里的中文原词，**不要换说法**：

| 编号 | doc_type（原词，直接填模板） | 适用场景 |
|---|---|---|
| 1 | 市场调研报告 | 行业 / 竞品 / 区域市场 / 评论 / 客户人群 / 价格 / 销量预测 等所有偏研究类产物 |
| 2 | 数据提取 | 仅交付原始数据明细（CSV / Excel / DB dump），无解读 |
| 3 | 广告数据与策略规划 | 广告投放数据 + 策略建议（合并交付，不拆开） |
| 4 | 工具开发 | 自研脚本 / 内部小工具 / Dashboard / 插件 等可执行产物 |
| 5 | 自动化流程 | RPA / 定时任务 / 流水线 / Workflow 等长期跑的自动化 |
| 6 | 数据核对 | 帮其他部门对账、口径校验、数据复核结论 |
| 7 | 文档翻译 | 中英 / 多语翻译稿件 |

**判定规则：**
- 用户已给明确类型 → 直接对到 7 项之一，不要新造词
- 用户描述模糊 → 给编号菜单让对方选；不要替对方拍板
- 若工作明显跨两类（例如"做了爬虫工具 + 给了一份数据"），让用户决定哪个是**主要交付物**，按主要那个填 doc_type，附带的在 Step 3 的「数据说明」/ 附件括号里说明

**这 7 项是组内对外契约，不要私自扩充。**真有新类别需要加入，先和组里对齐，更新本 SKILL.md 后再用。

### Step 3：约束核心结论的写法

- 一条一行，前缀 `- `（标准 markdown bullet）
- 数量 **1–8 条**；超过 8 条让用户挑最关键的，不要硬塞
- 每条 ≤ 2 句话；**不要在 bullet 里再嵌套子 bullet**（IM 渲染会乱）
- 写**结论**不写过程：
  - ✅「日本市场 Wi-Fi 6 路由器集中于 BUFFALO + NEC，合计市占 78%」
  - ❌「我们爬取了 200 条评论并做了分类」
- 数字带单位与对照口径（YoY / QoQ / 行业均值），不孤立
- 避免空话：「数据可观」「值得关注」「有待提升」

### Step 4：套模板，**只输出消息文本**

最终回复中**只**给出最终的发布消息正文，不要附带"以下是模板""你可以这样发"等说明语；用户要发到群里直接复制即可。

## 标准模板（中文，默认）

```
你好，这是 GL.iNet 数据团队对于需求序号 {ticket_id} {ticket_title} 的{doc_type}，请查收。{requester}

核心结论：
- {bullet 1}
- {bullet 2}
- {bullet ...}

如有任何问题或后续追加诉求，欢迎随时找数据组复核~
```

### 填好的样例

```
你好，这是 GL.iNet 数据团队对于需求序号 DT-2026-0427-01 日本 Wi-Fi 6 路由器竞品分析的市场调研报告，请查收。@张三

核心结论：
- 日本 Wi-Fi 6 路由器市场 BUFFALO + NEC 合计市占 78%，第三名 IO-DATA 仅 6%
- 中位价位 ¥18,800（约 ¥130 元），GL.iNet MT-3000 当前定价低于市场中位 32%
- 评论高频差评集中在「设置复杂」「日文界面缺失」，TP-Link 在这两项投诉率为 BUFFALO 的 1.6 倍
- 行业 Q1 同比 +12%，6E 渗透率仅 9%，6 仍是主力价位带

如有任何问题或后续追加诉求，欢迎随时找数据组复核~
```

## 变体（按场景微调）

### A. doc_type = 数据提取

把「核心结论：」改为「数据说明：」，bullet 写：
- 时间范围
- 数据来源
- 主要字段
- 数据量
- 注意事项 / 已知口径限制

```
数据说明：
- 时间范围：2026-01-01 ~ 2026-04-26
- 来源：AMC + Helium10 抓取
- 主要字段：ASIN / 7d_units / 7d_revenue / ad_spend
- 行数：12,438
- 注意事项：广告归因为 7-day click，自然销量未拆分
```

### B. doc_type = 文档翻译

把「核心结论：」改为「翻译说明：」，bullet 写：
- 原文 / 译文语言
- 字数 / 段落数
- 术语保留策略（保留原词 / 直译 / 意译）
- 不确定点 / 需要业务方确认的术语

### B2. doc_type = 工具开发 / 自动化流程

把「核心结论：」改为「交付说明：」，bullet 写：
- 入口 / 部署位置（脚本路径 / Dashboard URL / 触发方式）
- 运行频率（一次性 / 手动 / 定时 cron / 事件触发）
- 主要能力（1–3 条用人话讲清楚做了什么）
- 依赖与权限（需要的账号、Token、网络环境）
- 维护责任 & 失败兜底（谁值守 / 出错怎么联系数据组）

### C. doc_type = 数据核对

把「核心结论：」改为「核对结论：」，bullet 写：
- 核对对象（哪两边的数据 / 哪份口径 vs 哪份口径）
- 一致 / 不一致条数 + 比例
- 主要差异原因（口径差异 / 时差 / 漏数 / 重复…）
- 建议的修复或采纳口径

### D. 结论涉及风险或负面信号

把结尾客气话替换为更明确的提示，让接收方知道要尽快看：

```
若上述结论与决策预期偏差较大，请尽快联系数据组确认数据口径或追加分析，避免基于偏差结论拍板。
```

### E. 有附件 / 多份产物

申请人可能要找多个文件，在 `{doc_type}` 后用括号注明：

```
...的市场调研报告（含 1 份 HTML 主报告 + 1 份 ASIN 明细 CSV），请查收。
```

## 反模式（这些情况不要做）

- ❌ 把「GL.iNet 数据团队」改成"数据组"、"DT 同学"、"我们" —— 对外消息一律保留全称
- ❌ 把「核心结论：」字段名改写 —— 字段名是检索锚点，不能动
- ❌ 多 @ 几位申请人时塞在同一行用空格分隔 —— 用全角逗号或换行隔开，IM 渲染才不会糊
- ❌ 把 bullet 写成段落 —— IM 会压缩成一坨，丢失结构
- ❌ 在「请查收」之前漏掉"的" —— 语病：「需求 #1284 XX 的市场调研报告」
- ❌ 把过程当结论：「我们做了 X / 抓取了 Y / 调用了 Z 接口」—— 这些是**过程**，不是核心结论
- ❌ 在最终回复里附带"以上是建议消息，你可以..." —— 用户要的是可直接复制的成品

---

## 一句话工作流摘要

> 收齐 5 个槽位 → 选 doc_type → 用 1–8 条结论性 bullet 填空 → **只输出最终消息文本**。
