# 章节组织范式 · content-structure.md

不同类型的报告有不同的"骨架"。这份文档列出 4 种常见报告类型的推荐章节结构，以及每种章节的长度建议。**规划报告大纲时读这个；写具体内容时切回 components.md / charts.md。**

---

## 原则：每份报告都要有一条"故事线"

读者打开一份报告，3 秒内要知道：
1. 这是什么报告？（标题）
2. 整体情况怎么样？（概览）
3. 有什么问题？（关键发现）
4. 该做什么？（行动建议）

如果读者读完了，要能复述这 4 个答案——否则章节结构就失败了。

---

## 骨架 1：月度 / 季度复盘（最常用）

**典型用例**：`boss_report_2026Q1/index.html` 就是这个骨架。

### 章节顺序

```
HEADER（封面）
  ├── Badge: "GL.iNet · 内部报告" / "GL.iNet · 月度复盘"
  ├── H1: "YYYY年MM月 XX 复盘报告"
  └── meta: 报告期 / 数据来源 / 归因窗口 / 撰写

SECTION 1: 整体概览
  ├── 3 个环形 KPI 卡（总指标 + 分品类达标）
  ├── warning insight（整体问题一句话）
  ├── divider
  ├── 4 个扁平 KPI 卡（总花费 / 分项 / 效率指标）
  ├── 2 个图表（构成饼图 + 站点表现）
  ├── insight（投放效率评价）
  ├── divider
  ├── 横向柱图（各区域达标率）
  ├── divider
  ├── 分组柱图（Router vs KVM × 区域）
  └── insight（关键问题定位）

SECTION 2: 漏斗分析（可选）
  ├── 描述（触达人数、总花费、整体 ROAS）
  ├── 4 层漏斗（Awareness → Interest → Intent → Purchase）
  ├── 每层有 user count + 主 KPI + 效率 + guardrail
  └── insight（漏斗瓶颈在哪层）

SECTION 3: 细分主题（basket pattern）
  ├── 子主题 1（品牌词 / 核心产品线）
  │   ├── basket-title + basket-meta
  │   ├── 3 个 highlight 卡（搜索量 / 曝光占有 / 购买占有）
  │   ├── 双 Y 轴趋势图
  │   └── insight
  ├── 子主题 2-5（其他产品线 / 关键词组 / 市场）
  │   └── [同样结构]

SECTION 4: 总结与行动
  ├── 描述（一句话引导）
  └── 2 列 summary grid（3 个结论 + 3 个行动）

FOOTER（脚注，一行）
```

### 长度建议

| 章节 | HTML 行数 | 说明 |
|---|---|---|
| HEADER | 20-30 | 几乎不变 |
| Section 1 概览 | 100-200 | 最重的一章，KPI + 多图表 |
| Section 2 漏斗 | 80-150 | 可选；没有漏斗数据就删 |
| Section 3 细分 | 每子主题 30-50 × N | N 通常 3-5，总 150-250 |
| Section 4 总结 | 40-60 | 精简 |
| FOOTER | 3-5 | 一行 |
| **总计** | **500-800 行 body** | 加上 830 行 CSS + ~600 行 JS = ~2000 行 |

## 骨架 2：专项分析（聚焦单一问题）

**典型用例**：`seeding_validation_2026Q1`、`multi_touch_path_2026Q1` 这类深挖单个课题的报告。

### 章节顺序

```
HEADER
  └── Badge: "GL.iNet · 专项分析"

SECTION 1: 背景与问题
  ├── desc 段：为什么做这个分析？
  ├── 2-3 个扁平 KPI 卡（问题的量化表述）
  └── warning insight（问题严重程度）

SECTION 2: 分析方法
  ├── ach-section-title：数据口径
  ├── 术语表（glossary）：定义关键概念
  └── ach-section-hint：样本、时间窗口、归因模型

SECTION 3: 关键发现
  ├── 3-5 个发现，每个发现：
  │   ├── 小标题（basket-title 样式）
  │   ├── 图表或数据表
  │   └── insight 解读
  └── 中间用 divider 分段

SECTION 4: 结论与建议
  └── summary grid（2 列：结论 + 行动）

FOOTER
```

### 长度建议

专项分析比月度复盘短、但密度高。总长度 400-600 行 body。

## 骨架 3：漏斗诊断（全链路分析）

**典型用例**：`ppc_funnel_2026Q1` 这类专门看漏斗转化的报告。

### 章节顺序

```
HEADER
  └── Badge: "GL.iNet · 漏斗诊断"

SECTION 1: 漏斗全景
  ├── desc：漏斗定义、时间窗口
  ├── 4 个扁平 KPI 卡（总触达 / 转化人数 / 花费 / 整体 ROAS）
  ├── 4-5 层完整漏斗（funnel-grid）
  ├── 每层间的 rate-badge 标注转化率
  └── warning/insight 标出最弱一层

SECTION 2: 瓶颈深挖
  ├── 针对最弱那一层的多维度拆解
  ├── 分组柱图（按渠道 / 按产品线）
  ├── asin-tbl 明细表（TOP 5 / BOTTOM 5）
  └── insight（根因假设）

SECTION 3: 对照策略
  ├── 对照组定义（时间对照 / 人群对照）
  ├── 双 Y 轴趋势图（before / after）
  └── insight（策略是否有效）

SECTION 4: 行动建议
  └── summary grid（按"改什么"分组行动）

FOOTER
```

### 长度建议

漏斗诊断通常中等长度 500-700 行。漏斗网格本身就很占版面，Section 1 是重点。

## 骨架 4：对比分析（A/B / 实验结果 / 同比环比）

**典型用例**：两个地区、两种策略、上线前后对比类报告。

### 章节顺序

```
HEADER
  └── Badge: "GL.iNet · 对比分析"

SECTION 1: 对照组设置
  ├── desc：对照组定义（A 是什么、B 是什么、如何配对）
  ├── ach-section-title + glossary：样本量、显著性阈值
  └── 2 个 ad-kpi-card 并排（A 组 vs B 组 基本信息）

SECTION 2: 关键指标对比
  ├── 3 组 basket-highlight-row（每组 3 张卡：A / B / Δ）
  │   组 1：转化率
  │   组 2：ROAS
  │   组 3：新客占比
  └── insight（胜负初判）

SECTION 3: 显著性判断
  ├── 分组柱图（A vs B × 多个指标）
  ├── warning insight（哪些差异是噪声 / 哪些可信）
  └── 如果有时间序列，双 Y 轴对比

SECTION 4: 结论
  └── summary grid：左列"哪个赢了" + 右列"下一步怎么推广"

FOOTER
```

## 通用的章节写作规则

### 每个 section 的开头三件套

```html
<div class="section">
  <div class="section-label">第 N 部分</div>    <!-- 必要：一眼知道位置 -->
  <h2>章节标题</h2>                             <!-- 必要：一句话讲清主题 -->
  <div class="desc">一两句话提要，说清楚范围和数据来源。</div>  <!-- 必要：读者上下文 -->
  <!-- ... 具体内容 ... -->
</div>
```

### 每个图表/表格后加 insight

如果你放了一张图但没给出任何结论，读者会疑惑"你让我看这个想表达什么？"。所以：
- 图/表后 → 一个 `.insight` 或 `.insight.warning`
- 1-3 句话，点明"关键是什么 / 为什么重要 / 下一步"
- `<strong>` 高亮关键数字

### 章节节奏建议

- 一个 section 里超过 3 个图表时，用 `<div class="divider"></div>` 分段
- 图表和图表之间不要挤，至少留一段描述 + 一个 insight
- 如果章节超过 300 行，考虑拆成两个 section

### "故事线"检查清单

每写完一个章节，自问：
- [ ] 读者知道这个章节在回答什么问题吗？
- [ ] 关键发现（1-3 个）是不是用 `.insight` / 加粗数字表达出来了？
- [ ] 图表是支撑发现的证据，不是装饰？
- [ ] 本章结论能衔接到下一章吗？

## 反模式（这些章节组织方式不要用）

- ❌ **把所有数据都堆在一章**：超过 400 行的章节读者会迷路；拆成多个 section
- ❌ **图表轰炸无解读**：连续放 5 个图表不加 insight，读者不知道该看什么
- ❌ **结论分散在各章**：结论统一放 Section 4，不要每章末尾都插"小结"然后最后又重复
- ❌ **用章节标题讲答案**：标题应该是"主题"（如"各区域销售达标情况"），不是"答案"（如"美国区拖累了整体"），答案放 insight
- ❌ **meta 信息塞进正文**：数据来源、归因窗口、时间范围放 HEADER 的 meta 和 FOOTER，不要在章节 desc 里重复
- ❌ **没有行动建议的报告**：只讲问题不说下一步，读者看完什么也做不了；至少 1 条可执行的行动

## 快速决策：选哪个骨架？

```
报告范围是周期性全面复盘？
  └─ yes → 骨架 1（月度/季度复盘）
  └─ no
      └── 聚焦单一课题深挖？
          └─ yes → 骨架 2（专项分析）
      └── 看整条转化链路？
          └─ yes → 骨架 3（漏斗诊断）
      └── 对比两组/两阶段？
          └─ yes → 骨架 4（对比分析）
      └── 都不是？
          └─ 和用户确认：是否需要混合结构，或这根本不是一份 HTML 报告
```
