# templates.md — 8 章节对外清爽报告骨架（A 专用 · Rev 2）

> 本文件供 **ticket-aligner** 在 Step 5 读取作为产出骨架。
>
> 这一份是**真正发给需求方看的**——不是组员的内部技术笔记。所有内部黑话要翻译成业务语言，时间用小时 h（与 capabilities.md 基线一致），急 / 精给两档。skill 按此填充，不要新增 / 改名 / 调序章节。

```
# 需求 {ticket_id} 评估说明

**禁止**在头部出现：`基于 capabilities.md 版本 vX` / `类型分类: 销量预估,广告ROI` / `🟢🟡🔴` 等内部标记。这些是组员看的，不是需求方看的。

## 章节一：我们的理解

```
## 一、我们的理解

{≤ 4 句话忠实复述需求方核心诉求 + 一句话点出"为什么需要这份评估"——即需求方的最终决策用途。}
```

### 写法约束
- ≤ 4 句话；用业务语言（不是数据组黑话）
- **不要**改写或扩展原意 —— 扩展放后续章节
- **必须**复述需求方明示的目的（"为了销量预估"、"为了拍板生产节奏"），这是后续工时和范围对齐的锚

### 反例 vs 正例
- ❌ "需求方想知道欧洲市场情况" —— 太空，扩成什么内容？
- ❌ 把诉求当 todo list 详细列 —— 丢了主次
- ✅ "由于 FCC 禁令，BE10000 暂时无法在美上市，您希望基于欧洲市场数据评估销量上限与广告投入产出，为生产节奏和销售预测模型提供输入。"

## 章节二：我们计划这样做

```
## 二、我们计划这样做

{2–4 段叙述，每段 2–4 句}

- 数据来源（**业务语言**：自家销售后台 / 公开评论 / 行业数据库 / 自家广告后台 / 用户社区讨论 等。**不写**工具名）
- 对照基线（如某时间窗 / 某些竞品清单 / 行业平均 等）
- 输出形态（**按场景推荐**：图表多 → HTML 报告；要离线归档 → 加导出 PDF；需求方要改稿 → 加导出 Word；数据明细多 → Excel + 一页摘要。详见 dimensions.md "输出形态"推荐判断）
```

### 写法约束
- **产出形态推荐要给理由**：不要只写"HTML 报告"，要写"考虑到本次涉及 8 张销量趋势图与价格分布对比，建议以 HTML 为主载体（图表交互保真），同时导出 PDF 用于归档"。1 句话理由足够，不要堆术语；推荐逻辑见 dimensions.md "输出形态"推荐判断
- **适度透露方法**：让对方感觉到工作量（"我们会爬取过去 12 个月的全量评论"）但不暴露工具（不写 "用 Apify Reddit Scraper"）
- 用动词 + 名词组合，不要堆 bullet：
  - ✅ "我们会盘点欧洲 5 个主要站点过去 12 个月的高客单价路由器销售数据，对照 Top 8 竞品的价格分布与销量趋势"
  - ❌ "数据源：SP-API + Helium10 + Keepa；范围：5 市场；时间：L12M；竞品：8 个"

### 业务语言对照表（强制使用）

| 内部黑话 | 业务语言 |
|---|---|
| SP-API | 自家销售后台 |
| AMC SQL | 自家广告投放后台 |
| Helium10 / Keepa | 行业数据库（销量与价格估算） |
| Apify Amazon 评论 | 公开评论数据 |
| Apify Reddit Scraper | 用户社区讨论（Reddit / 论坛） |
| Discourse API | 用户社区讨论（GL.iNet 论坛 / OpenWrt 论坛） |
| LLM 语义分析 | 评论内容分类与情感识别 |
| 11 维度情感分析 | 多维度用户口碑全景图 |
| 关键词词典标注 | 主题归类与标注 |

## 章节三：时间预估

```
## 三、时间预估

我们建议两个时间档位，可根据您的紧急程度选择。下面把工时按工作块拆分，便于您看到时间花在哪里。

### 精准版（推荐）：约 X–Yh

| # | 工作块 | 主要内容 | 工时（h） |
|---|---|---|---|
| 1 | {块名 1} | {1–3 句业务语言描述} | 约 {n}h |
| 2 | {块名 2} | {1–3 句} | 约 {n}h |
| ... | ... | ... | ... |
|  | **合计** |  | **约 X–Yh** |

- 优势：{为什么值得这个时间 —— 例：销量预估置信区间约 ±15%}
- 占比说明：约 1/{x} 工时花在 {块 A}，约 1/{y} 在 {块 B}（让需求方一眼看到预算重心）

### 快速版（如急用）：约 X'–Y'h

| # | 工作块 | 与精准版的差异 | 工时（h） |
|---|---|---|---|
| 1 | {块名 1} | {精简点} | 约 {n}h |
| ... | ... | ... | ... |
|  | **合计** |  | **约 X'–Y'h** |

- 会牺牲的：{**明确量化** —— 例：置信区间从 ±15% 放大到 ±35%；样本量从 N 降到 N/3；不做敏感性分析}
- ⚠️ {仅"不确定性高"的类型加这一行：决策时建议综合多方信息}
```

### 写法约束
- **精准版 ≥ 60h 的 ticket 必须用工作块表格拆分** —— 4–8 行 + **合计行**；让需求方看到时间花在哪里，避免"为什么要这么久"的疑问。表格列固定：序号 / 工作块名（3–6 字短名）/ 主要内容（1–3 句业务语言）/ 工时（h，"约 Xh"）
- **快速版同步表格化** —— 把"主要内容"列改为"与精准版的差异"，让两档对比一目了然
- 表格之后必加 1 句**占比说明** —— 指出工时占比最高的两块（如"约 1/3 工时在维度权重提取，约 1/3 在销量预估"），让需求方看到预算重心
- **工作块粒度建议 4–16h** —— 单块 > 16h 必须再拆子块（避免"一块一天"的粗粒度遮蔽真实步骤）；< 4h 可考虑合并相邻块（避免行数过多与噪声）
- **工时估算来自 capabilities.md 的真实任务基线** —— 不要在每块基础上再整体加 buffer；合计 ±20% 范围呈现已经足够；多市场扩展时考虑核心模型可复用（"4h 核心 + N × 2h 适配"），不要简单线性乘 N
- **中型 ticket（精准版 30–60h）可不用表格** —— 但必须按"覆盖 / 优势 / 牺牲"三段式叙述并量化差异
- **小 ticket（精准版 ≤ 30h）单档即可** —— 表格非必需，参见下方"单档场景"
- **两档之间必须有可比较的具体差异** —— "快速版会粗糙一点"太空，必须量化（"置信区间从 ±15% 放大到 ±35%"、"样本从 1500 条降到 500 条"、"不做敏感性分析"等）
- "精准版（推荐）"标题里**必须**带"推荐"二字 —— 防止需求方默认选快速版贪便宜
- 工时数字按 capabilities.md 中的 h 基线 + 加 ±20% 范围呈现（如 28h → "约 28–34h"）；表格内每行 h 相加应等于合计行；必加"约"字（h 数也是估算不是承诺）
- 表格行的"工作块名"用业务语言（"竞品发现与价格盘点"、"用户规格维度权重提取"），不出工具名

### 反例 vs 正例
- ❌ "精准版约 130–150h，覆盖 6 市场全量..." —— 没拆，需求方看不到时间花在哪，会疑问"为什么要这么久"
- ✅ 工作块表格 6 行 + 合计行；占比说明明示"维度权重提取占 1/3，销量预估占 1/3"
- ❌ 表格行的"工作块名"含"评论挖掘 + Apify Reddit 抓取"（暴露工具）—— 用"用户规格维度权重提取"
- ❌ 表格行内含"约 1 天 / 整天 / 半天" —— 对外用 h（与 capabilities.md 基线一致），禁止用"天 / 工作日"
- ❌ 单工作块写"约 30h 评论挖掘 + 维度提取 + 词典扩词 + 反向校准"（块塞太多，遮蔽真实步骤）—— 拆成 2–3 个独立块，每块 4–16h
- ❌ 单工作块只写"约 1h 起 ETL 任务" —— 太碎，可与相邻块合并
- ❌ 6 市场 × 6h 销量预估 = 36h（简单线性乘）—— 用"4h 核心模型 + 6 × 2h 适配 = 16h"更接近真实工作量

### 单档场景（小 ticket / 可靠性高的类型）

可靠性高的类型（评论挖掘 / 自家广告 ROI 等）可以**只给单档**：

```
## 三、时间预估

预计约 {X}h 完成。

- 覆盖：{...}
- 输出：{...}

> 如急用可压缩到 {Y}h，但因 {具体理由} 不太建议。
```

## 章节四：关于精度的说明

```
## 四、关于精度的说明

{1–2 段话解释"为什么有些分析急用会粗糙"。需要让需求方理解时间投入与精度的关系——这是教育对方时间成本的核心载体。}
```

### 写法约束
- **只在两档版**出现 —— 单档不必有这一节
- 用 1–2 段话讲清"为什么急用会粗糙"，不要泛泛而谈
- 针对 ticket 主类型给具体说明：
  - **销量预估**："销量预估依赖至少 12 个月的历史数据 + 类比品交叉验证 + 模型校准。校准时间不够会让置信区间显著放大；快速版只能给类比法粗估。"
  - **市场容量**："市场容量需要交叉多个数据源做口径校验。少做一轮校验，估算误差容易从 ±20% 扩大到 ±40%。"
  - **用户研究**："用户偏好需要足够样本量稳定结论。样本量减半时，强弱结论之间的差距可能被噪声淹没。"
  - **广告 ROI**：可靠性较高，通常不需要这一节（用单档代替）

## 章节五：需要您确认的几点

```
## 五、需要您确认的几点

> 以下几个细节请您在回复中告知，每问回数字即可。

1. {问题简述}
   - A) {选项}（约 +0h）
   - B) {选项}（约 +6h）
   - C) {选项}（约 -3h，但精度会下降）
   - D) 您来定，建议 {字母}

2. ...
```

### 写法约束
- ≤ 6 条 —— 超过让需求方决策疲劳
- 每选项标"约 ±Xh"（与 capabilities.md 基线 / dimensions.md 工时弹性一致；必加"约"字）
- 必有"D) 您来定，建议 X" 兜底
- 不用开放式问答（"您希望颗粒度多细？"）—— 必须给 ABC 候选
- "h 代价"直接来自 dimensions.md 各类必问维度的工时弹性：
  - 弹性 ≥ 1h → 直接标"约 +Xh"
  - 弹性 < 1h → 标"几乎无影响"或"约 +0h"
- 跨品类 ticket 弹性 × 1.5 后呈现（对外**不暴露**这个倍数）
- 不要在这章出现内部黑话（"AMC 拉不动"、"凭证过期"）

## 章节六：关于范围的几点说明

```
## 六、关于范围的几点说明

{委婉指出做不了 / 不合理的项}
{措辞：}
- "考虑到 {X} 的限制，目前我们的工具体系下无法稳定提供 {Y}，建议改用 {Z} 方式作为参考。"
- "{某项} 在我们的现有数据中只能获取到 {低精度版本}，是否仍需要纳入？建议 {建议}。"
- "{某项} 通常需要 {前置条件}，建议在 {时间点} 提前 {动作}，避免事后无法补救。"

{如果全部能做：}
> 以上诉求经评估均可处理，无需调整范围。
```

### 写法约束
- **每个委婉指出后必须附"建议改用 X"** —— 只软不指出会让需求方对范围产生错误期望
- 不直接说"做不了"或"无法实现"
- 如果全部能做也要明示一句话（避免对方漏读这一节，误以为我们跳过了）
- 跨部门 / 跨产品线的限制（如"未提前打点的归因事后不可补救"）要在这一章用业务语言说出来

### 委婉表达范本

| 场景 | 委婉表达 |
|---|---|
| 缺数据授权 | "考虑到第三方品牌的广告后台无法直接访问，{X} 数据我们目前无法稳定提供。建议改用 Y 方式（公开数据 + 估算）作为参考。" |
| 跨产品外推 | "{X 产品} 的数据可作为参考，但与 {Y 产品} 不能等价外推；建议在结论中明确标注'参考性外推'，避免决策时误用。" |
| 未上市新品预估 | "由于 {产品} 尚未上市，预估置信度会低于已上市产品。建议在 Phase 1 出基础估算，Phase 2 上市后用真实数据校准。" |
| Dashboard / 工具开发 | "这部分需要工程立项（独立的开发投入），我们暂不在此评估范围内，建议另起需求。" |

## 章节七：下一步

```
## 七、下一步

- 收到您的反馈后，我们会在约 3h 内合并 + 启动分析

### 写法约束
- 反馈合并预计**用小时 h**（如"约 3h"，不写"半天"或"0.5 个工作日"——与全文单位一致）
- 同步渠道写明确——"会在群里同步"太空，要写"在钉钉数据组群 / 在某个 doc"等（GL.iNet 数据组当前默认协作平台为钉钉；如组织变更可在本文件中替换）
- 末尾"如有其他想法..."这一句**必有**——给需求方一个开放出口

## 八、附录：信息来源

- 1 段话 / 3–6 行 bullet

- 仍然用业务语言，不出工具名
- **小 ticket 可省略此章** —— 体量轻时不必把信息源也展示一遍

## 文档结尾

```
> 如果您选择"快速版"或对范围有调整，我们也会相应缩减分析深度，确保如期交付。
```

末尾这一句是**软提示**——让需求方意识到选择是有代价的，但不会让他们感觉被压力。可选；小 ticket 可省。

|---|---|

| `{ticket_title}` | 工单标题，与原文一致 |

## 长度控制

- 大 ticket（综合需求）：1.5–2 页 A4 / 600–900 字
- 小 ticket（单一功能 / 简单对比）：1 页内 / 300–500 字（章节四 + 八可省）
- **不要**写成 5 页技术文档——会让需求方"使用门槛感"上升

## 章节自适应规则

| ticket 体量 | 章节适配 |
|---|---|
| 巨型（≥ 60h） | 全 8 章节；时间预估两档 + 工作块表格**强制** |
| 大 / 综合（24–60h） | 全 8 章节；时间预估两档；工作块表格推荐但不强制 |
| 中（8–24h） | 全 8 章节，但章节八可省；时间预估可选两档（不需要表格） |
| 小（≤ 8h，全量 ≤ 6h） | 4–5 章节（一、二、三 单档、五、七）；省略 章节四 / 六 / 八 |

### 章节边界硬约束（Rev 2.1）

- **alignment.md 严格 8 章节为限** —— 章节自适应只允许**减少**（小 ticket 可省 4 / 6 / 8），**不允许新增**任何章节
- **禁止追加任何"给组员看"的内容** —— 包括但不限于：反馈合并指引 / 组员注意事项 / 内部 SOP / 接单后续动作清单。这类组员可见信息归 SKILL.md Step 7 终端输出，**不进文件**
- alignment.md 是**完全发给需求方的对外文档**，与组员的内部回路（终端输出 / decomp.md）严格分文件
- 末尾"软提示"句（"如果您选择'快速版'..."）不算新增章节，可保留

## 总结：对外报告与对内 Plan 的差异（提醒 skill）

|  | 对外（本 skill 输出） | 对内（ticket-decomposer 输出） |
|---|---|---|
| 时间单位 | 小时 h（"约 18h"） | 小时 h（"5h"） |
| 任务标识 | 自然段落叙述 | U1 / U2 / ... 编号 |
| 可行性 | 软措辞（"目前无法稳定提供"） | 🟢🟡🔴 三色 |
| 数据源 | 业务语言（"自家销售后台"） | 工具名（"SP-API" / "Apify"） |
| 字段 | 概括 + 范例 | DoD / 输入 / 输出 / 工具 / 风险 |
| 长度 | 1.5–2 页（短小精悍） | 5–10 页（详尽可执行） |

**核心原则**：对外报告是**告诉需求方我们在做什么 + 为什么需要这个时间**；对内 Plan 是**告诉组员怎么做 + 谁拿什么单元**。两份文件目标读者完全不同，写法不混用。
