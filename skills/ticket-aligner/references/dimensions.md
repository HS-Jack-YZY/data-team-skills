## 1. 市场容量

### 必问维度
- **地理范围（国家 / 区域）** —— 每多 1 个市场 +2h（Keepa / SP-API），+3h（评论抓取）。默认：Top 5 站点（US/UK/DE/FR/IT 或 US/DE/UK/FR/ES，取决于品类）
- **时间窗** —— L12M ↔ L6M 几乎等价；YoY 对比 +1h。默认：L12M
- **价位带定义** —— 含/不含税、页面价 / 实付价混用 +0.5h（口径切换）。默认：含税页面价
- **类目定义** —— Amazon BSR 节点 / 自定义关键词集合。自定义关键词集合首次定义 +2h。默认：BSR 节点
- **"市场容量"口径** —— 销量件数 / 销售额 / 在售 ASIN 数。三选一 +0h；全部要 +3h。默认：销量件数 + 销售额双口径
- **数据源** —— Keepa / Helium10 / 行业报告。前两者 ±30% 误差；行业报告延迟 6–12 月

### 已知坑
- 🟡 未上市新品的预估必须降置信度
- 🟡 跨品类时基线 ×1.5（参考 capabilities.md 跨品类节）

## 2. 竞品分析

### 必问维度
- **竞品清单** —— 需求方提供 / 数据组 AI 抓取 / 两者结合。AI 抓取 +3h（含人工校验）。默认：需求方给清单 + 数据组补充 5 个长尾
- **平台范围** —— Amazon / Best Buy / Newegg / 独立站 / 多平台。每多一个平台 +4h
- **对标维度** —— 价格 / 规格 / 销量 / 评分 / 评论维度权重。规格类 +2h；评论维度权重 +6h（依赖 11 维度情感）。默认：价格 + 销量 + 评分 + 5 个核心规格
- **市场范围** —— US / EU / 多市场。每多一市场 +6h
- **时间窗** —— L12M / L6M / 当前快照。当前快照 -2h（不需要历史）
- **市场份额来源** —— Helium10 估算 / 销量倒推 / 行业报告。三选一 +0h

### 已知坑
- 🔴 竞品广告投放数据不可达
- 🟡 第三方品牌真实利润 / ROI 不可达，只能用毛利估算

---

## 3. 用户研究

### 必问维度
- **数据源组合** —— Amazon 评论 / Reddit / GL.iNet 论坛 / OpenWrt 论坛 / SNBForums / 问卷。每多一个 +0.5–2h（依 capabilities）。默认：Amazon 评论 + Reddit + GL.iNet 论坛
- **目标产品 / 竞品** —— 自家 1 个 / 自家 ≥ 3 / 含竞品。每多一个 ASIN ≈ +0.5h
- **市场范围** —— US / DE / UK / JP / 多市场。每多一市场 ≈ +1h（评论抓取）
- **关键词词典** —— 复用 7 套现有 / 新建。复用 +0h；扩词 +2h；新建一套 +4h。默认：复用 + 扩词
- **情感分析颗粒度** —— 三色（正/中/负）/ 五分（强烈正→中→强烈负）/ 多维度热力图。多维度 +6h
- **样本量底线** —— 单产品 ≥ X 条评论。默认：≥ 200 条；< 100 条要降置信度
- **输出形态** —— 用户画像（细分人群比例）/ 痛点排行 / 升级意愿。每加一种 +2h

### 已知坑
- 🟡 评论稀疏的产品（如新品上市前期）样本不足，需 Reddit / 论坛补
- 🟡 跨语言评论（DE / JP）需要语种处理 +1h

---

## 4. 广告 ROI

### 必问维度
- **归因窗口** —— 7d-click / 14d-click / view-through。多窗口对比 +1h。默认：7d-click
- **归因层级** —— campaign / ad group / ASIN / 全店。每深一级 +1h。默认：ASIN 级
- **是否拆 Sponsored Products / Brands / Display** —— 拆分 +2h（**仅自家**有授权）
- **时间粒度** —— 日 / 周 / 月。日级 +1h
- **是否含品牌词 / 非品牌词** —— 拆分 +1h
- **核心指标选择** —— ACoS / TACoS / CTR / CVR / CPC / ROAS。全要 +0h；前提是 AMC 凭证可用
- **跨产品外推假设** —— 是 / 否。是 → 必须在第 3 章假设清单中**明确标记** + 第 5 章降置信度（🟡）

### 已知坑
- 🔴 竞品 Sponsored Brands vs Products 拆分（缺授权）—— 第 5 章必须标 🔴
- 🟡 BE3600 → BE10000 这类外推**仅作参考**

---

## 5. 价格策略

### 必问维度
- **价格口径** —— 含税 / 不含税；页面价 / 实付价 / 折后价。默认：含税页面价 + 折后价双口径
- **市场范围** —— US / EU 单 / EU 多市场。每多一市场 +2h
- **价格分布颗粒度** —— Top 100 BSR / 全市场 / 自定义价位带。Top 100 +0h；全市场 +3h
- **历史窗口** —— 当前快照 / L6M / L12M。L12M +1h
- **价格弹性分析** —— 是否做（销量 vs 价格回归）。是 +3h
- **竞品定价对照** —— 是否对照。是 +2h

### 已知坑
- 🟡 折扣 / 促销期价格波动需在数据清洗时排除
- 🟡 不同币种（€ / £ / $）汇率换算口径需明确

---

## 6. 销量预估

### 必问维度
- **预估对象** —— 已上市产品 / 未上市产品 / 系列。未上市必须 🟡 + 标注置信度
- **市场范围** —— 单市场 / 多市场（线性加）
- **时间窗** —— 上市 90 天 / 6 月 / 12 月。每多一档 +2h
- **预估方法** —— 类比法（同价位带均值）/ 回归法（自家历史）/ 加权（含广告 ROI 输入）
- **置信区间** —— 是否给 80% CI / 90% CI。是 +1h
- **敏感性分析** —— 价格 ±10% / 广告投入 ±20% 的销量影响。是 +3h

### 已知坑
- 🟡 BE10000 / BE14000 这类未上市新品**置信度低**，必须明确告知
- 🟡 跨品类（如摄像头）置信度更低

---

## 7. 产品定义

### 必问维度
- **决策点清单** —— ID / 网口 / USB / 存储 / 屏幕 / 天线 等。每多一个决策点 +1h（关键词词典）
- **数据来源** —— 评论 / Reddit / 问卷 / 销售数据。问卷需要时间，建议至少 2 周提前期
- **目标用户分群** —— 极客 / 远程办公 / 小企业 / 旅行者。多分群 +2h
- **决策颗粒度** —— 二选一（A vs B）/ 偏好排序 / 加权权重。加权权重 +3h
- **市场差异** —— US / EU / JP 偏好对比。每多一市场 +2h

### 已知坑
- 🟡 决策点过多（>5 个）易稀释信号，建议先聚焦核心 3 个

---

## 8. 促销 / 运营策略

### 必问维度
- **覆盖促销类型** —— BD / LD / DOTD / Coupon / Promo Code 全选 vs 部分。每多一类 +2h
- **市场数** —— US / UK / DE / CA / AU 全 5 个 vs 子集。每多一市场 +2h
- **输出层级** —— 成效复盘 / 全年排程 / 选品策略 / 利润测算。每多一层 +3h
- **跨部门联动** —— 独立站 / 社媒 / 站外帖。每多一渠道 +3h
- **利润测算口径** —— 毛利 / 净利 / 含/不含广告费 / 含/不含运营人力。每深一层 +1h
- **时间窗** —— 单次活动 / 季度 / 全年。全年 +6h
- **CD（cooldown）规则** —— 是否分析"同一商品两次大促之间需间隔多少天"。是 +2h

### 已知坑
- 🔴 历史未打点的跨部门归因事后追溯不可达
- 🟡 行业基准（其他品牌的促销节奏）只能靠公开抽样

---

## 9. 复盘 / 后验分析

### 必问维度
- **复盘对象** —— 单次活动 / 单季度 / 单产品线。单活动 → 整季度 +6h
- **复盘维度** —— 总体 / 分市场 / KPI / 工作项 / 异常事件（参考春促复盘的五大维度）。每多一维度 +1.5h
- **归因层级** —— campaign / channel / KOL / station / SKU。每深一级 +2h
- **是否提供改善建议** —— 仅复盘 vs 复盘 + 后续策略。加策略建议 +4h
- **数据预先打点情况** —— 这是约束项不是弹性项。**未打点的归因事后不可补救**，必须在第 5 章明确告知

### 已知坑
- 🔴 跨部门未埋活动 ID 的转化来源事后无法定位
- 🟡 库存周转 / 利润数据需要跨财务系统对齐

## 10. 工具 / Dashboard

> ⚠️ **识别后建议拆分立项**：本类 ticket 通常包含两块——**数据需求**（属本 skill 范围）+ **工具开发**（不属）。第 6 章 MVP 必须明确这一拆分。

### 必问维度（仅覆盖数据需求部分）
- **看板的指标范围** —— 核心 KPI 数 + 拆分维度数。每多 5 个指标 +2h
- **数据源是否需要打通** —— 已有报表 vs 需要新建 ETL。新建 ETL +8h 起
- **刷新频率** —— LIVE / 小时级 / 日级。LIVE +6h（架构）
- **用户权限** —— 公开 / 限定团队 / 角色化。角色化 +3h
- **可更新的目标值（target）** —— 是否要支持运营手动更新 target。是 +2h

### 已知坑
- 🔴 看板的工程实现 / 长期维护**不归本流程**
- 🟡 LIVE 看板的稳定性依赖 SP-API / AMC 接口稳定性

## 通用维度（任何类别都要问）

### 交付时间
- **截止日期** —— 决定 MVP vs 全量
- **是否赶节点**（如某次大促前）—— 是 → 砍 Phase 2，只做 Phase 1

### 输出形态
- **HTML 报告 / .docx / .pptx / Excel / Dashboard / 数据集**
  - 应用 html-report 模板 ≈ +0.5h
  - 自定义 PPT ≈ +3h（设计 + 排版）
  - 仅数据集（CSV / Excel）≈ -1h（无解读）
- 默认：HTML 报告（用 html-report skill 模板）

### 接收人
- 决策者（C-level / VP）—— 输出更精炼，TL;DR 章节加重
- 执行层（运营 / PM）—— 输出更细节，附数据明细

### 后续可追加分析的开放度
- 一次性交付 vs 可迭代。一次性 → 强收敛 MVP；可迭代 → Phase 1 + Phase 2

## 工作日换算指引（Rev 2 · 给 skill 看的）

> 本文件中的工时弹性都以 **h 为单位**——这是组员推算成本的内部尺度。但 ticket-aligner 的输出（章节五、章节三）必须**对外用工作日**呈现。换算规则如下：

### 基本换算

- **6h = 1 工作日**（按典型组员每日有效工作时数计；与 ticket-decomposer 共用同一系数，保证两 skill 工时一致）
- 章节三"时间预估"的 X / Y 工作日数：
  - capabilities.md 推算总 h ÷ 6 得工作日
  - 加 ±20% 范围呈现（如 18h → "约 3–4 工作日"），单档可取中位（"约 3 工作日"）
  - **必加"约"字**——工作日是估算不是承诺

### 章节五选项工作日代价的近似规则

| h 弹性 | 工作日呈现 |
|---|---|
| +6h 及以上 | "约 +1 工作日" 起 |
| +3h ~ +5.9h | "约 +0.5 工作日" |
| +1.5h ~ +2.9h | "约 +0.5 工作日"（同上一档；不细分到 +0.25 工作日避免噪声） |
| +0.5h ~ +1.4h | "几乎无影响" 或 "约 +0 工作日" |
| -1h 及以下 | "约 -0.5 工作日" 或更长（同正向） |
| -3h 及以上 | "约 -1 工作日" 起 |

### 跨品类 ticket 换算

跨品类 ticket（非路由器品类）所有相关 h 弹性**先 ×1.5**再换工作日。换算结果对外**不要暴露**这个倍数（不写"由于跨品类 ×1.5 倍"）——直接给最终工作日数。

### 示例

dimensions.md 写："地理范围每多 1 个市场 +2h（Keepa）+ 3h（评论抓取）"
- 内部组员看到：单市场切换总 +5h
- skill 在章节五呈现：`B) 加 1 个市场（约 +1 工作日）`
- 跨品类 ticket（如安防摄像头）：5h × 1.5 = 7.5h → 章节五呈现 `约 +1 工作日`（不暴露倍数）

### 反模式

- ❌ 在对外报告里写"约 +5h" / "约半天" / "约 3 个小时" —— 必须用"工作日"
- ❌ 工作日精度细到 ±0.25 工作日 —— 噪声大于信号；用 0.5 工作日颗粒度
- ❌ 不加"约"字 —— 让需求方误以为是承诺
- ❌ 在对外报告暴露跨品类 ×1.5 倍数 —— 内部考量不出现在对外文档
