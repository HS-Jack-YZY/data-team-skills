# ticket-aligner — 需求 ticket 对外对齐报告生成器

把"需求方丢过来的 ticket"转换成"组员可一眼判定可行性 + 可发给需求方推动对齐的结构化文档"，并附带一条**给组员私发需求方的钉钉确认私信**，把对方拉回报告评审形成闭环。

整套数据组 ticket 处理 pipeline：

```
ticket → ticket-aligner（首次调用）→ alignment_v1.md → 发给需求方
                                  ↓
                             需求方反馈
                                  ↓
       ticket-aligner（再次调用，喂入反馈）→ alignment_v2.md（… vN）
                                  ↓
                ticket-decomposer → plan.md → 组员并行执行
```

每次调用本 skill 都自动产出新版本号：首次写 `alignment_v1.md`，已存在 v1 时写 v2，依此类推。历史版本永远保留——这是给需求方做溯源 + 内部审计的依据。

| `@` 引用 | `@ticket-aligner 帮我对齐这个 ticket` |

**触发后 skill 会一次性问 4 件事**（缺啥补啥；首条提问里已经写过的就跳过）：

## 核心契约（Rev 2 · 不要违反）

1. **8 章节顺序与名称固定** —— 一、我们的理解 → 二、我们计划这样做 → 三、时间预估 → 四、关于精度的说明 → 五、需要您确认的几点 → 六、关于范围的几点说明 → 七、下一步 → 八、附录：信息来源（可选）。空段保留标题。
2. **整份报告完全是给需求方看的** —— **不出现** 🟢🟡🔴 / Tx 编号 / capabilities.md 引用 / 工具缩写（Apify / Keepa / AMC / SP-API / Helium10 等）。所有内部黑话翻译为业务语言，对照表见 `references/templates.md`。
3. **时间单位对外用"小时 h"** —— 与 capabilities.md 的 h 基线一致，避免换算误差。两档预估时呈现"约 X–Yh"或单值"约 Xh"；"约"字必加（h 数是估算不是承诺）。
4. **"急 vs 精"两档必须给出**（不确定性高的类型） —— 销量预估 / 市场容量 / 未上市新品 / 跨品类 必拆两档；评论挖掘 / 自家广告 ROI 可单档。两档之间**必须量化差异**（"置信区间从 ±15% 放大到 ±35%"），不空写"会粗糙一点"。
5. **精准版默认推荐** —— 标题里**必须**带"（推荐）"二字，防止需求方默认选快速版贪便宜导致输出粗糙引发不满。
6. **第六章软措辞 + 必给替代提议** —— 不直说"做不了"，用"目前我们的工具体系下无法稳定提供 X，建议改用 Y"。如果全部能做也明示一句"以上诉求经评估均可处理"。
7. **第五章选项 h 代价** —— 每选项标"约 ±Xh"（与 capabilities.md / dimensions.md 工时弹性一致）；必有"D) 您来定，建议 X" 兜底；不用开放式问答。
8. **跨品类 ticket（非路由器）工时基线 ×1.5** —— 对外**不暴露**这个倍数（直接给乘后的最终 h）。
9. **小 ticket（全量 ≤ 6h）不出"急/精"两档** —— 单档够了；可省略章节四 / 六 / 八。
10. **Dashboard / 工具开发类 ticket 在第六章建议拆分立项** —— 用业务语言："这部分需要工程立项，我们暂不在此评估范围内，建议另起需求"。
11. **报告长度** —— 大 ticket 1.5–2 页（约 600–900 字）；小 ticket 1 页内（300–500 字）。**不要**写成 5 页技术文档。
12. **必须同时输出对齐确认私信** —— Step 7 终端四件套之第 (d) 件，按 `references/templates.md` "附加产物：对齐确认私信" 段生成。私信 5–8 行，**不复读**报告内容，必须明确"请在文档里直接评论 / 修改"，必须给截止天数 + 默认推进策略 + 文件路径。这是把对方拉回报告评审、形成闭环的关键，不是可选项。

## 工作流（用户触发后按序执行）

### Step 1 — 输入捕获

让用户提供 ticket 全文（或路径）+ 元信息（需求序号、提需求方、接单组员）。

如果用户**只**给了一句话描述但没有原文，**必须**索要原文 —— 第 1 章"我们听到的是"依赖忠实复述，没有原文容易漂移。

### Step 2 — 类型分类（多选）

把 ticket 归入以下 10 类的一类或多类。多类 ticket 后续叠加额主问题。

| 类别 | 触发关键词速览 |
|---|---|
| 市场容量 | "市场容量"、"市场规模"、"销量占比"、"客单价" |
| 竞品分析 | "竞品"、"对标"、"vs"、"市场占比"、"评分" |
| 用户研究 | "用户偏好"、"画像"、"评论分析"、"痛点"、"使用场景" |
| 广告 ROI | "广告"、"ACoS"、"TACoS"、"CTR"、"CVR"、"投放"、"种草" |
| 价格策略 | "定价"、"售价"、"价格带"、"价格敏感度" |
| 销量预估 | "销量预估"、"销量预测"、"上限"、"市场空间" |
| 产品定义 | "产品定义"、"硬件配置"、"减配"、"系列定位" |
| 促销 / 运营策略 | "BD"、"LD"、"DOTD"、"Coupon"、"促销"、"全年规划" |
| 复盘 / 后验分析 | "复盘"、"回顾"、"达成"、"归因"、"事后分析" |
| 工具 / Dashboard | "看板"、"Dashboard"、"LIVE"、"自动化"、"工具" |

详细识别规则与各类必问维度见 `references/dimensions.md`。

### Step 3 — 读 capabilities.md

```
Read references/capabilities.md
```

加载工时基线 + 可行性锚点到上下文。这是后续所有 h 数与 🟢🟡🔴 标注的来源。

### Step 4 — 读 dimensions.md 中匹配类型的章节

```
Read references/dimensions.md
```

只关注 Step 2 命中的类别章节 + 通用维度章节 + 默认值速查表。**不要全读**。

### Step 5 — 读 templates.md

```
Read references/templates.md
```

加载 8 章节骨架 + 占位符约定 + 写法约束。

### Step 6 — 生成对外清爽报告（8 章节）

按 templates.md 的骨架填充 8 章节。**所有内部黑话翻译为业务语言**（参考 templates.md 中的"业务语言对照表"）。关键提醒：

- **章节一、我们的理解**：≤ 4 句话忠实复述 + 一句话点出需求方决策用途
- **章节二、我们计划这样做**：2–4 段叙述。**适度透露方法**让对方感觉到工作量，但不暴露工具名（用"自家销售后台"而不是"SP-API"）
- **章节三、时间预估**：两档（精准版 推荐 / 快速版 如急用）。每档说明"覆盖 + 优势 / 牺牲"。**不确定性高**的类型必给两档；可靠性高的类型可单档
- **章节四、关于精度的说明**：仅两档版有；1–2 段话讲清"为什么急用会粗糙"——这是教育对方时间成本的核心载体
- **章节五、需要您确认的几点**：≤ 6 条编号问答，每选项标"约 ±Xh"（与 capabilities.md 基线一致），必有"D) 您来定，建议 X" 兜底
- **章节六、关于范围的几点说明**：软措辞 + 必给替代提议；如全部能做也明示一句"以上诉求经评估均可处理"
- **章节七、下一步**：反馈合并工时（约 3h）+ 同步渠道 + 对接组员
- **章节八、附录：信息来源**（可选）：1 段话 / 3–6 行 bullet，业务语言；小 ticket 可省

**自适应规则**：

| ticket 体量 | 章节适配 |
|---|---|
| 巨型（≥ 60h） | 全 8 章节；时间预估两档 + **工作块表格强制**（与 templates.md "章节自适应规则"一致） |
| 大 / 综合（24–60h） | 全 8 章节；时间预估两档；工作块表格推荐但不强制 |
| 中（8–24h） | 全 8 章节，可选两档；章节八可省 |
| 小（≤ 8h，全量 ≤ 6h） | 4–5 章节（一、二、三 单档、五、七）；省略 章节四 / 六 / 八 |

> 体量分档（巨型 / 大 / 中 / 小）在本文件、`references/templates.md` "章节自适应规则" 与 "截止天数建议" 三处保持单一来源；如需调整请同步三处。

### Step 7 — 落盘 + 终端四件套

#### 7.1 编号 ambiguity gate（先于任何写入）

调用本 skill 时若**未含编号**（既无短编号 `66` 也无长编号 `DT-2026-0427-01`），停下问：

> 请提供 ticket 编号（短编号如 `66`，或完整编号如 `DT-2026-0427-01`），我才能创建 ticket 目录。

无编号一律不写文件——避免在错位置或假名字下产出。这条规则的存在是因为：ticket 目录名是编号 + slug 的拼接，编号是稳定锚点，没有它后续 ticket-decomposer / html-report 都对不上号。

#### 7.2 派生 ticket 目录名

ticket 目录名 = `<编号>_<slug>`：
- `<编号>`：用户给的短编号或长编号原样保留
- `<slug>`：从需求标题派生，kebab-case（保留首字母大写词形如 `Manual-SFT1200-Translation-FR-DE-ES-PL`）

例：`66_Manual-SFT1200-Translation-FR-DE-ES-PL`、`DT-2026-04-28-01_BE7200-User-Research`

#### 7.3 落盘几何（ticket 目录内）

```
<编号>_<slug>/
└── docs/
    ├── ticket.md                     # 原 ticket 归档（用户粘贴的纯文本拷贝）
    └── alignment_docs/
        ├── alignment_v1.md           # 本 skill 首次输出
        ├── alignment_v2.md           # 反馈后再次调用产出
        └── alignment_v{N}.md         # 后续反馈轮次
```

#### 7.4 自动版本号（首次 / 反馈轮次）

在写入前用一行 shell 决定版本号——**不要靠 LLM 推理**当前是不是反馈轮次：

```bash
# 在 ticket 目录的父级运行
TICKET_DIR=<编号>_<slug>
mkdir -p "$TICKET_DIR/docs/alignment_docs"

# 算下一个版本号：找已有最大 N，+1；没有则 1
NEXT=$(ls "$TICKET_DIR/docs/alignment_docs/"alignment_v*.md 2>/dev/null \
       | sed -E 's/.*alignment_v([0-9]+)\.md/\1/' \
       | sort -n \
       | tail -n 1)
NEXT=$((${NEXT:-0} + 1))
echo "$TICKET_DIR/docs/alignment_docs/alignment_v${NEXT}.md"
```

然后用 `Write` 写入这个路径。

如果原 ticket 是用户粘贴的纯文本（没有路径来源）且 `docs/ticket.md` 不存在，同时归档：

```
Write <编号>_<slug>/docs/ticket.md
```

历史版本不要覆盖：v1 / v2 / ... 全部保留，需求方溯源依赖它。

#### 7.5 终端四件套

落盘后在对话里**必须**输出"终端四件套"（不写任何文件，仅在对话呈现）：

**(a) 文件路径** —— 让用户能立刻打开 / IM 转发。

**(b) 第 5 章节选** —— 章节五"需要您确认的几点"是组员可直接精简后转发需求方的核心问答，按 ABCD 选项简列即可（每问一行带推荐项）。让用户不打开文件就能预览要发出去的内容。

**(c) 组员 SOP** —— 仅在终端呈现，**不写进 alignment 文件**。提示组员后续如何处理该 ticket，按当前 ticket 实际情况删改以下默认 4–5 条：

- 章节五任意问题未回复按推荐项默认；时间档位（最末问）若未回复默认精准版；**超过 (d) 私信中给定的截止天数（巨型 3–5 天 / 大 2–3 天 / 中 1–2 天 / 小 1 天）后按此推荐项默认推进**，与 (d) 私信"逾期按推荐项默认"承诺闭环
- 需求方反馈合并后产出 `alignment_v2.md`，再由 ticket-decomposer 拆 `decomp.md`
- 跨品类 ticket 工时倍数（×1.5）是否生效（仅非路由器品类生效；本 ticket 适用 / 不适用 写明）
- 章节五中如有"需协调外部资源"项（问卷投放渠道、新平台接入、跨部门埋点等），提示组员在 v2 之前先 outreach；若确认无法协调，回退到推荐项
- 如 ticket 含 Dashboard / LIVE 工具部分，提示工具开发部分单独拆分立项

**(d) 对齐确认私信** —— 按 `references/templates.md` "附加产物：对齐确认私信" 段生成的钉钉/IM 私发文本，组员复制后**点对点私发**给需求方（不要群发）。私信里**不复读**报告内容，只承担"指路 + 强制行动"：必含 @需求方 + 一句话定位 + 最关键的 2–3 个点（时间预估 / 第五章问题数 /（如有）第六章范围调整提醒）+ "请直接在文档里评论或修改"指令 + 截止天数与默认推进策略 + 本次 alignment_v{N}.md 路径（即刚刚落盘那份）。截止天数按体量动态选（巨型 3–5 天 / 大 2–3 天 / 中 1–2 天 / 小 1 天）。这是把对方拉回报告评审、形成闭环的关键产物，不是可选项。

**反模式提醒**：**不要**把整份 8 章节文档复读到对话里（已落盘，给路径即可）；**不要**把组员 SOP 或对齐确认私信文本写进 alignment 文件（污染对外文档），仅终端输出；**(d) 私信本身也不要复读 alignment 文件内容**（私信只承担"指路 + 强制行动"，详见 `references/templates.md` "对齐确认私信"反模式）。

## 反模式（Rev 2 · 这些情况不要做）

- ❌ 触发条件放宽到自然语言（"用户提到 ticket 就触发"）—— 仅显式 slash / @ 命令
- ❌ 报告里出现 🟢🟡🔴 / Tx 编号 / capabilities.md 引用 / 工具名（Apify / Keepa / AMC / SP-API / Helium10 等）—— **必须**翻译为业务语言
- ❌ 用"工作日"/"天"/"半天"作对外时间单位 —— 改用小时 h（与 capabilities.md 基线一致，避免换算误差与"一块一天"的粗粒度）
- ❌ 章节一改写或扩展原意 —— 复述要忠实，扩展放后续章节
- ❌ 直接说"不能做"、"无法实现"、"做不到" —— 必须委婉 + 给替代提议
- ❌ "急 vs 精"两档之间没有可比较的具体差异（"快速版会粗糙一点"太空）—— 必须量化
- ❌ "急 vs 精"做成 3 档以上 —— 决策疲劳；最多两档
- ❌ "精准版"标题里没有"（推荐）"二字 —— 防止需求方默认贪便宜选快速版
- ❌ 章节五写开放式问答（"您希望颗粒度多细？"）—— 必须给 ABC 选项 + h 代价 + "D) 您来定" 兜底
- ❌ 章节六软措辞但只指出问题不给替代 —— 让需求方对范围产生错误期望
- ❌ 章节六全部能做时跳过这一节 —— 必须明示一句"以上诉求经评估均可处理"，避免对方漏读
- ❌ 跨品类 ticket（如酒店安防摄像头）直接套用路由器工时基线 —— 工时基线 ×1.5 后呈现，对外不暴露这个倍数
- ❌ 小 ticket（全量 ≤ 6h）强行拆"急/精"两档 —— 单档够了；省略章节四 / 六 / 八
- ❌ Dashboard / 工具开发类 ticket 强行做完整数据分析评估 —— 在章节六建议拆分立项（业务语言）
- ❌ 报告写成 5 页技术文档 —— 大 ticket 1.5–2 页，小 ticket 1 页内
- ❌ 在最终回复里把整份报告复读 —— 已落盘，给路径即可
- ❌ 在 alignment_v{N}.md 末尾追加 8 章节之外的任何内容（如"反馈合并指引"、"组员注意事项"、"内部 SOP"段、"对齐确认私信文本"）—— 这类组员可见信息只在 Step 7 终端输出，不进文件；alignment_v{N}.md 严格 8 章节为限，是**完全发给需求方的对外文档**
- ❌ Step 7 终端输出只给路径不给章节五节选 / 组员 SOP / 对齐确认私信 —— 终端四件套缺一不可
- ❌ 对齐确认私信复读 alignment 文件内容（把章节一二三摘要全抄进去）—— 私信只承担"指路 + 强制行动"，复读会让对方两次阅读同样内容，反而降低读报告意愿；详见 `references/templates.md` "附加产物：对齐确认私信"
- ❌ 把对齐确认私信和 delivery-message 混为一谈 —— 前者是对齐阶段"请您审"的**点对点私发**，后者是交付完成"这是结果"的**群发**，目的、渠道、内容都不同，不互相替代

## 文件结构

```
skills/ticket-aligner/
├── SKILL.md                       # 你正在读的这个文件
├── references/
│   ├── capabilities.md            # 数据源 / 工具清单 + 工时基线（与 ticket-decomposer 共享）
│   ├── dimensions.md              # 10 类需求的额主问题清单 + 工时弹性 + 默认值
│   └── templates.md               # 8 章节骨架 + 写法约束 + 占位符
└── assets/
    ├── example_BE7200.md          # 大 ticket 完整示例（与 research_flow.md 对照）
    └── example_USB_tethering.md   # 小 ticket 收敛形态示例
```

落盘根目录约定（ticket 目录由本 skill 在首次调用时创建；后续 skill 都往同一目录里加文件）：

```
<编号>_<slug>/                                  # ticket 根，由本 skill 首次调用时创建
├── docs/                                       # 过程性文档
│   ├── ticket.md                               # 原 ticket 归档（本 skill 落盘）
│   ├── alignment_docs/
│   │   ├── alignment_v1.md                     # 本 skill 首次输出
│   │   ├── alignment_v2.md                     # 反馈后再次调用本 skill 产出
│   │   └── alignment_v{N}.md                   # 后续反馈轮次
│   ├── plan.md                                 # ticket-decomposer 输出
│   ├── data/                                   # 中间数据（social-reviews-analyzer 等写入）
│   └── delivery.md                             # delivery-message 镜像归档
└── <编号>_<slug>.html                          # html-report 最终成品（在 ticket 根，不在 docs 下）
```

> 把"过程文档"放 `docs/` 下、"对外成品"（html）放 ticket 根，是数据组的工作流约定——一眼区分中间产物和可对外发的交付物。

## 一句话工作流摘要

> 收齐 4 个槽位 + 编号（无编号必停下问）→ 派生 ticket 目录 `<编号>_<slug>` → 读 capabilities + dimensions + templates → 按 8 章节骨架填，每项标 h → 一行 shell 算 v{N} 版本号 → 落盘 alignment_v{N}.md → 在对话里给终端四件套（路径 + 章节五节选 + 组员 SOP + 对齐确认私信）。
