# decomposition_recipes.md — 按需求类型拆分配方（B 专用）

> 本文件供 **ticket-decomposer** 在 Step 3 读取，用于按 alignment_v2 的"类型分类"决定典型拆分序列。
>
> 每个配方给出：**推荐流程**（4–5 阶段）+ **典型 U 序列**（引用 work_unit_patterns.md 中的 P1–P14）+ **依赖图**。skill 在生成 Plan 第 2、3、4 章时引用本文件。多类别 ticket → 配方合并取并集，并去除重复 U。

---

## 通用四步流程框架

无论哪类需求，都按"收 → 标 → 算 → 写"四步组织：

```
1. 收数据   → P1–P7（按数据源拆并行）
2. 打标签   → P8–P10（清洗、词典标注、LLM 语义）
3. 做统计   → P11–P12（专题分析）
4. 给建议   → P13（章节撰写）
```

复盘类追加阶段 0：跨部门埋点对账（P14）。

## 配方 1：用户研究

**推荐流程**：收数据 → 打标签 → 做统计 → 给建议 → 报告撰写

**典型 U 序列**：

```
U1  Reddit 抓取（关键词×N）           P1     1h × N      并行
U2  Amazon 评论抓取（自家+竞品 ASIN）  P2     0.5h × M    并行
U3  GL.iNet 论坛 + OpenWrt 论坛抓取    P3     0.5h × K    并行
U4  词典扩词（如有新决策点）           P9     2–4h        与 U1–U3 并行（基于抽样）
U5  词典标注                          P8     0.5h        依赖 U1+U2+U3+U4
U6  LLM 语义分析                      P10    1h / 1000条  依赖 U5
U7  统计与可视化（按问题切）           内嵌    1h / 问题    依赖 U6
U8  章节撰写（HTML 报告 N 章）         P13    1.5h × N    依赖 U7
```

**依赖图**：

```
U1 ─┐
U2 ─┤
U3 ─┼─→ U5 ─→ U6 ─→ U7 ─→ U8
U4 ─┘
```

**典型工时（中等规模）**：18–32h；2 人并行 3 工作日。

## 配方 2：竞品分析

**推荐流程**：收竞品 + 自家数据 → 价格 / 销量分析 → 多维对标 → 输出对标矩阵

**典型 U 序列**：

```
U1  Keepa 数据拉取（竞品 ×K + 自家 ×M）  P5     0.2h × (K+M)   并行
U2  Amazon 评论抓取（同上）              P2     0.5h × (K+M)   并行
U3  竞品规格抓取（人工 / Helium10）      自定义  0.5h × K        并行
U4  价格分布分析                        内嵌    2h / 市场       依赖 U1
U5  词典标注 + 11 维度情感分析           P8+P11  8h             依赖 U2
U6  对标矩阵汇总（评分 + 规格 + 价格）    内嵌    3h             依赖 U3+U4+U5
U7  章节撰写                            P13    1.5h × N        依赖 U6
```

**依赖图**：

```
U1 ──→ U4 ─┐
U2 ──→ U5 ─┼→ U6 ─→ U7
U3 ────────┘
```

**典型工时**：单市场 ≈ 16h；多市场（×N）线性加。

## 配方 3：市场容量

**推荐流程**：拉销量 / 价格 / BSR → 容量测算 → 增长率 → 报告

**典型 U 序列**：

```
U1  Keepa Top 100 BSR 全字段（每市场）   P5     3h / 市场      并行
U2  SP-API 自家销量（每市场）            P7     0.3h / 市场    并行
U3  Helium10 类目估算（每市场）          P5 变体  1h / 市场      并行
U4  价位带聚合 + 容量测算                 内嵌    2h / 市场      依赖 U1+U3
U5  增长率（YoY 对比）                   内嵌    1.5h / 市场    依赖 U1（多时间点）
U6  章节撰写                            P13    1.5h × N        依赖 U4+U5
```

**典型工时**：单市场 ≈ 10h；5 市场 ≈ 30h（U1–U3 并行可压到 18h 关键路径）。

## 配方 4：广告 ROI

**推荐流程**：拉 AMC 数据 → 自家分层（campaign / ASIN）→ TACoS 趋势 → 预算建议

**典型 U 序列**：

```
U-pre  AMC 凭证验证                    自定义  0.5h         前置
U1   AMC SQL：基础指标（ACoS/CTR/CVR）  P6     1h          并行
U2   AMC SQL：归因层级（ASIN / 创意）    P6     2h          并行
U3   AMC SQL：TACoS 趋势                P6     1h          并行
U4   SP-API 销量（关联广告归因）          P7     1h          并行
U5   预算分配建议（Sponsored Brands vs Products） 内嵌  3h   依赖 U1+U2+U3+U4
U6   章节撰写                           P13    1.5h × N    依赖 U5
```

**依赖图**：

```
U-pre → U1 ─┐
       U2 ─┤
       U3 ─┼→ U5 → U6
       U4 ─┘
```

**注意**：跨产品外推（如 BE3600 → BE10000）不另起 U，但在 U5 的输出中**必须**标"参考性外推 · 置信度低"。

**典型工时**：≈ 11h；1–2 人 2 工作日。

## 配方 5：价格策略

**推荐流程**：抓价格分布 → 价格弹性 → 竞品对照 → 定价建议

**典型 U 序列**：

```
U1  Keepa 历史价格（自家 + 竞品）       P5     0.2h × N      并行
U2  独立站后台价格 / 折扣数据           P7 变体 1h / 市场     并行
U3  价格分布分析                       内嵌    2h / 市场     依赖 U1
U4  价格弹性回归（销量 vs 价格）        内嵌    3h            依赖 U1+U2
U5  竞品定价对照表                      内嵌    2h            依赖 U1
U6  定价建议（含敏感性）                 内嵌    2h            依赖 U3+U4+U5
U7  章节撰写                           P13    1.5h × N      依赖 U6
```

**典型工时**：≈ 12h（不含弹性回归则 9h）。

## 配方 6：销量预估

**推荐流程**：历史数据 → 类比 / 回归建模 → 敏感性 → 输出

**典型 U 序列**：

```
U1  Keepa 历史（自家 + 类比品 ×K）      P5     0.2h × (1+K)   并行
U2  SP-API 自家历史                    P7     1h            并行
U3  广告 ROI 输入（如有，从配方 4 复用）  P6     1h            并行
U4  销量预估建模                        P12    4–6h           依赖 U1+U2+U3
U5  敏感性分析                         内嵌    1.5h          依赖 U4
U6  章节撰写                           P13    1.5h × N      依赖 U5
```

**典型工时**：基础 ≈ 10h；含广告输入 ≈ 12h；多市场 / 多产品线性加。

**未上市产品**（BE10000 / BE14000 类）：U4 的输出**必须**标"低置信度，需上市后校准"。

## 配方 7：产品定义

**推荐流程**：评论 + 论坛挖掘 → 决策点偏好统计 → 报告

**典型 U 序列**：

```
U1  Reddit + 论坛抓取(决策点关键词)     P1+P3    0.5–1h × N   并行
U2  Amazon 评论抓取（自家 + 同价位竞品）P2      0.5h × M    并行
U3  词典标注（决策点对应词典）          P8       0.5h × Y    依赖 U1+U2
U4  LLM 语义分析（情感 + 偏好聚类）     P10      1h / 1000条 依赖 U3
U5  决策点统计（每个决策点出一份偏好表） 内嵌     1h × 决策点  依赖 U4
U6  章节撰写                          P13     1.5h × N     依赖 U5
```

**典型工时**：3 决策点 ≈ 14h（参考 example_BE7200 Phase 1）；多决策点线性加。

## 配方 8：促销 / 运营策略

**推荐流程**：拉历史促销数据 → 跨市场 / 跨类型对比 → 全年规划

**典型 U 序列**：

```
U1  SP-API 拉每市场每促销类型历史成效   P7      1h / 市场     并行
U2  独立站 + 站外帖 历史（如有）        P7 变体  1h / 渠道     并行
U3  AMC 关联广告与促销期归因            P6      1h            并行
U4  各市场 BD/LD/DOTD 成效复盘表        内嵌     1.5h / 市场   依赖 U1
U5  跨市场对比 + 5 市场组合规划         内嵌     6h            依赖 U4
U6  全年 12 个月组合排程建议            内嵌     4h            依赖 U5
U7  章节撰写                          P13     1.5h × N      依赖 U6
```

**典型工时**：5 市场全年规划 ≈ 22–28h；3 人并行约 1.5 工作日（关键路径 U5+U6 串行 10h）。

## 配方 9：复盘 / 后验分析

**推荐流程**：跨部门埋点对账 → 拉数据 → 五大维度归因 → 改进建议

**典型 U 序列**（以单一活动为例）：

```
U-pre0  跨部门活动埋点对账（独立站 / 社媒 / 站外帖） P14   2h × 部门  并行
U1      SP-API 销售数据导出                       P7    1h       并行
U2      AMC 广告数据导出                         P6    1h       并行
U3      库存周转数据                             P7 变体 0.5h     并行
U4      五大维度归因（总体 / 分市场 / KPI / 工作项 / 异常）内嵌  1.5h × 5  依赖 U-pre0+U1+U2+U3
U5      改进建议汇总                             内嵌   4h        依赖 U4
U6      章节撰写                                P13   1.5h × N    依赖 U5
```

**典型工时**：单活动 5 市场全维度 ≈ 16h；季度复盘 ≈ 20h。

**警告**：U-pre0 必须**前置**——如果跨部门未在活动期打点，本配方在事后**不可执行**，回退到 ticket-aligner 出 alignment_v3 重谈范围。

## 多类别合并规则

ticket 横跨多类时（例：BE10000 = 市场容量 + 广告 ROI + 销量预估），按以下规则合并：

1. **取所有相关配方的 U 序列并集**
2. **去重**：同名 U（如多个配方都有 P5 Keepa 拉取）合并为一个
3. **重新拉依赖图**：合并后某些 U 同时被多条配方依赖，更新到第 4 章依赖图
4. **小心工时叠加陷阱**：U 合并后总工时**不应**简单相加——避免重复计算
5. **整体不超过 alignment_v2 第 6 章 MVP 的总 h（容差 ±10%）**：超出 → 提示用户重新对齐范围

---

## 工时与 alignment 一致性自检

skill 在生成 Plan 后**必须**做一次自检：

```
Σ(所有 U 的预计 h) ≈ alignment_v2 第 6 章 MVP 总 h
容差：±10%
```

不一致 → **不要**自行抹平：停下，输出"工时不一致警告"，提示用户检查 capabilities.md / 维度变更 / 漏估的 U。

容差范围内（±10%）：在 Plan 第 5 章工期估算的注释里说明差异来源（缓冲 / 协调 / QA）。
