# work_unit_patterns.md — 典型工作单元模板（B 专用）

> 本文件供 **ticket-decomposer** 在 Step 4 读取，作为生成第 3 章工作单元清单时的模板池。
>
> 每个 U-pattern 是一个"原子工作单元"的填空模板，含字段：输入 / 输出产物 / 工具 / 基线 / DoD / 风险。skill 在生成 Plan 时挑选适用的 U-pattern，复制并填空，**不要**新造无 pattern 的自由格式工作单元（除非真的没合适的 pattern——此时基线必须标 `（耗时待估）`）。
>
> **每个工作单元 ≤ 8h**。如果某 pattern 默认 > 8h（例：MT6000 大量评论抓取≈ 2h × 多市场会超），按市场 / 关键词进一步拆分。

## P1. Reddit 评论抓取

```
- 单元名：Reddit 评论抓取（{子版块清单} ×{N} · {关键词清单} ×{M}）
- 输入：
  - 关键词清单（≤ 5 关键词 / 批；超过拆多 U）
  - 目标子版块清单（如 r/HomeNetworking, r/TravelHacks）
- 输出产物：reddit_<keyword>_<YYYYMMDD>.csv（8 列：正文、标题、社区、投票数、时间、URL、子版块、抓取批次）
- 推荐工具：Apify Reddit Scraper
- 预计耗时：{N×M ≤ 5 时 1h；多批次按 1h/批}
- 依赖：词典（如果新建）；否则无
- 可并行：与 P2 / P3 / P4 完全并行
- DoD：
  - 关键词清单覆盖 100%
  - 清洗后行数 ≥ 100（提及关键词的内容）
  - 归档到 data/raw/reddit/<ticket-id>/
- 风险：限流（每批 ≤ 5 关键词）、冷门关键词无结果（需提前做 1 关键词试跑）
```

## P2. Amazon 评论抓取

```
- 单元名：Amazon 评论抓取（{ASIN 数} ASIN · {市场}）
- 输入：
  - ASIN 清单
  - 市场（US / DE / UK / JP 等）
- 输出产物：amazon_reviews_<asin>_<market>_<YYYYMMDD>.csv
  （字段：品牌、型号、评分、标题、正文、日期、市场）
- 推荐工具：Apify Amazon 评论（SHADER 代理 + Playwright）
- 预计耗时：普通 ASIN 0.5h / 个；评论数 > 2000 的热门 ASIN 2h / 个；多市场 ×N（线性）
- 依赖：ASIN 清单
- 可并行：与 P1 / P3 / P4 完全并行
- DoD：
  - 全部 ASIN × 市场 抓取完成
  - 单 ASIN 单市场行数符合预期（普通 ≥ 200，热门 ≥ 1000）
  - 字段完整无缺
  - 归档到 data/raw/amazon/<ticket-id>/
- 风险：热门 ASIN 大量抓取需要拆分批次；多市场需注意语种（DE / JP 评论需后续翻译标注）
```

## P3. 论坛抓取（GL.iNet 论坛 / OpenWrt 论坛）

```
- 单元名：{论坛名} 论坛抓取（{N} 搜索词）
- 输入：搜索词清单
- 输出产物：forum_<source>_<keyword>_<YYYYMMDD>.csv
- 推荐工具：forum_scraper.py（基于 Discourse API）
- 预计耗时：0.5h / 搜索词
- 依赖：搜索词清单（来自词典或 PM 提供）
- 可并行：与 P1 / P2 / P4 完全并行
- DoD：搜索词覆盖 100%；归档到 data/raw/forum/<source>/<ticket-id>/
- 风险：API 速率限制（一般不会触发）；论坛活跃度低导致结果稀疏
```

## P4. SNBForums 抓取（Playwright）

```
- 单元名：SNBForums 抓取（{N} 搜索词）
- 输入：搜索词清单
- 输出产物：snbforums_<keyword>_<YYYYMMDD>.csv
- 推荐工具：Playwright MCP（无 API，浏览器自动化）
- 预计耗时：1.5h / 搜索词（含人工筛选）
- 依赖：搜索词清单
- 可并行：与 P1 / P2 / P3 并行（但占用浏览器，注意冲突）
- DoD：搜索词覆盖 100%；筛选后保留高质量帖（≥ 5 投票或带技术分析的）
- 风险：UI 改版导致选择器失效；建议优先级低（先看 P3 的覆盖度，不够再补）
```

## P5. Keepa 价格 / 销量拉取

```
- 单元名：Keepa 数据拉取（{N} ASIN · {市场}）
- 输入：ASIN 清单
- 输出产物：keepa_<asin>_<market>_<YYYYMMDD>.json（含历史价格、BSR、销量估算）
- 推荐工具：keepa_api.py
- 预计耗时：0.2h / ASIN；Top 100 BSR 全字段 ≈ 3h
- 依赖：ASIN 清单 + Keepa API token
- 可并行：与 P1–P4 / P6–P7 完全并行
- DoD：全部 ASIN 历史数据已落地；缺失 ASIN 列入 issue list
- 风险：销量估算误差 ±30%；token 配额每月有限，注意分配
```

## P6. AMC SQL 查询

```
- 单元名：AMC SQL 查询（{指标}）
- 输入：SQL 模板 + 时间窗 + 归因层级
- 输出产物：amc_<query_name>_<YYYYMMDD>.csv
- 推荐工具：AMC SQL 控制台（Claude 协助 SQL）
- 预计耗时：1h / SQL（含取数 + 验证）；创意维度 2h
- 依赖：AMC 凭证 + SQL 模板
- 可并行：与 P1–P5 / P7 完全并行
- DoD：
  - SQL 跑通无错
  - 关键指标值与上次基线对比合理（<20% 偏差）
  - 数据落地到 data/amc/<ticket-id>/
- 风险：凭证过期（单独建一个 P-pre 验证凭证）；查询超时（超过 5 分钟需要拆 SQL）
```

## P7. SP-API 销量 / 报表导出

```
- 单元名：SP-API 数据导出（{范围}）
- 输入：ASIN / 市场 / 时间窗
- 输出产物：spapi_<scope>_<YYYYMMDD>.csv
- 推荐工具：SP-API client
- 预计耗时:单 ASIN 单市场 0.3h；全店报表 1h；多市场 3h
- 依赖：SP-API 凭证（自家 ASIN）
- 可并行：与 P1–P6 完全并行
- DoD：所有目标 ASIN × 市场 × 时间窗 已导出
- 风险：API 速率限制；部分子站点凭证需要单独申请
```

## P8. 关键词词典标注（复用既有词典）

```
- 单元名：关键词词典标注（{词典名} ×{N}）
- 输入：上游抓取数据（CSV）+ 既有词典
- 输出产物：labeled_<dataset>_<YYYYMMDD>.csv（追加列：tag_id_design / tag_ports / tag_usb 等）
- 推荐工具：Python 脚本（复用 clean_reddit_data.py 风格）
- 预计耗时：0.5h / 数据集（应用 1–3 套词典）
- 依赖：上游抓取数据
- 可并行：可与不同数据集的 P8 并行
- DoD：
  - 标注覆盖率 ≥ 60%（即 ≥ 60% 的行被至少一个词典命中）
  - 命中率 < 30% 时提示需要扩词或新建词典
  - 落地到 data/labeled/<ticket-id>/
- 风险：词典命中率低（提示走 P9）；多义词误命中（需人工抽检 ≥ 50 条）
```

## P9. 关键词词典扩词 / 新建

```
- 单元名：词典 {名} {扩词 / 新建}
- 输入：领域词种子（PM 提供 / 评论抽样）
- 输出产物：dict_<name>_v<X>.json（结构：positive_words / negative_words / scenarios / aliases）
- 推荐工具：Python + LLM 协助生成词候选 + 人工筛选
- 预计耗时：扩词 2h / 套；新建一套 4h
- 依赖：领域语料（≥ 50 条评论 / 帖子）
- 可并行：与抓取并行（用前期抽样语料训词典）
- DoD：
  - 词数 ≥ 30（新建）/ 或 +20%（扩词）
  - 在 100 条人工标注集上召回率 ≥ 60%、准确率 ≥ 70%
  - 词典版本号 + 变更说明落地
- 风险：人工筛选耗时漂移；建议组员协作 review
```

## P10. LLM 语义分析

```
- 单元名：LLM 语义分析（{数据集} ×{条数}）
- 输入：标注后或抓取后的数据集
- 输出产物：llm_<dataset>_<YYYYMMDD>.csv（追加列：sentiment / theme / persona）
- 推荐工具：Claude API（按 token 计费）
- 预计耗时：1h / 1000 条；人工校准（≥ 100 条）+ 2h
- 依赖：上游数据集
- 可并行：与统计步骤前的不同数据集 P10 并行
- DoD：
  - 全部数据有 sentiment 标签
  - 抽 100 条做人工 vs LLM 一致性 ≥ 80%（不达标走 prompt tuning，再做一次）
  - 落地到 data/analyzed/<ticket-id>/
- 风险：长文本超 token 限制（需分块）；某些类目 prompt 漂移（先用小批量 50 条试调）
```

## P11. 11 维度情感分析

```
- 单元名：11 维度情感分析（{品类} 全品牌）
- 维度：WiFi 性能 / 稳定性 / 设置 / 性价比 / VPN / 固件 / 信号 / USB / 设计 / 端口 / 品牌力
- 输入：全量已标注 + LLM 语义分析后的数据
- 输出产物：sentiment_matrix_<category>_<YYYYMMDD>.csv（行：品牌 × 维度，值：正负面比例）
- 推荐工具：Python + Pandas + matplotlib / ECharts
- 预计耗时：8h（关键词覆盖好的情况下；含可视化）
- 依赖：P8 + P10 完成；P9 词典覆盖度足够
- 可并行：无（这是综合分析，靠串行）
- DoD：
  - 11 维度均有覆盖（即每个维度 × 品牌交叉至少 ≥ 30 条评论支撑）
  - 输出热力图 + 数据明细表
  - 异常值（极小样本）单列附录
- 风险：维度间不独立（如"信号"和"WiFi 性能"重合）—— 需要在 prompt / 词典层做去重；样本少的品牌（< 100 条）置信度低，附录单列
```

## P12. 销量预估建模

```
- 单元名：销量预估建模（{产品} · {市场}）
- 输入：Keepa / SP-API 历史数据 + 类比产品 + 价格点
- 输出产物：sales_forecast_<product>_<market>.xlsx（月度预测 + 80% CI）
- 推荐工具：Python + sklearn / 简单回归
- 预计耗时：单产品单市场 4h（基础）；含广告 ROI 输入 6h
- 依赖：P5 + P7 完成
- 可并行：不同市场 / 不同产品的 P12 完全并行
- DoD：
  - 输出 12 个月预测 + 80% CI
  - 含敏感性分析（价格 ±10% / 广告 ±20%）
  - 验证集 RMSE 与基线相比未恶化
- 风险：未上市产品（如 BE10000 / BE14000）置信度低，必须在结论中标注；跨品类基线不适用
```

## P13. 章节撰写（HTML）

```
- 单元名：章节撰写「{章节名}」
- 输入：上游统计 / 可视化输出 + 报告骨架
- 输出产物：<编号>_<slug>/<编号>_<slug>.html 的 {章节名} 章节（或带 assets 时 <编号>_<slug>/index.html 的 {章节名}）
- 推荐工具：html-report skill（templates + 组件库）
- 预计耗时：1.5h / 章节（含图表 + 解读）；整份报告 8h
- 依赖：P10 / P11 / P12 等分析步骤完成
- 可并行：不同章节可由不同组员并行写
- DoD：
  - 章节内容与上游数据一致
  - 图表使用 html-report 的 ECharts 工厂函数
  - 配色 / 字体 / 间距遵循 html-report 设计系统
- 风险：图表数据手工搬运易出错（建议直接 import 数据 JSON）
```

## P14. 跨部门归因对账（复盘类专用）

```
- 单元名：跨部门活动埋点对账（{部门 / 活动}）
- 输入：自家数据（SP-API / AMC）+ 跨部门提供的活动 ID / 埋点
- 输出产物：attribution_<event>_<YYYYMMDD>.xlsx（活动期间归因明细）
- 推荐工具：Python + Excel
- 预计耗时：2h / 部门对接（含跨部门 IM 来回）
- 依赖：跨部门提供的活动埋点（前置约束）
- 可并行：不同部门可并行
- DoD：
  - 归因覆盖率 ≥ 80%（剩余 ≤ 20% 归"自然 / 未知"）
  - 主要差异点 ≤ 3 个（差异 > 3 个需要二次对账）
- 风险：**活动期间未打点的归因不可补救**（capabilities.md 已注明）；跨部门响应慢
```

## 通用 DoD（所有 U 适用）

每个工作单元交付时都需满足：

- ✅ 输出文件落地到约定路径
- ✅ 数据行数 / 字段 / 范围与单元名声明一致
- ✅ 异常 / 缺失 / 限流情况记录到 `data/<ticket-id>/issues.md`
- ✅ 在 IM 抢单贴下 ✅ 标注完成（可选附路径）

---

## 单元颗粒度与拆分原则

- **单 U 上限 8h**：超过就按维度拆。例：P11（8h）已到上限；如果跨多市场，拆成"P11-US（8h）+ P11-DE（8h）+ ...".
- **单 U 下限 0.5h**：低于 0.5h 不值得单列；并入相邻单元。
- **依赖必须显式**：U 之间的依赖 / 并行关系**必须**在 Plan 第 4 章依赖图中体现。
- **不分配人名**：U 不带 owner 字段——分配在 IM 抢单环节决定。
