1. **使用场景** — 这位用户用产品做什么? (家庭主用 / 备份 / 移动 / 商业 / 其他 / 未提及)
2. **用户类型** — 技术水平 / 角色身份 (新手 / 普通用户 / 进阶爱好者 / 专业人员 / 决策者 / 未提及)
3. **生态偏好** — 对品牌 / 平台的态度 (深度绑定某品牌 / 多品牌尝试 / 无明显偏好 / 未提及)

5. **当前痛点或目标** — 一句话, 必须源自评论原文
6. **决策阶段** — (选购中 / 已购买使用 / 安装故障中 / 观望 / 评价他人 / 未提及)
</task>

<rules>
【首要区分】你描述的是写 <comment_to_analyze> 这条评论的作者.
- 不是原帖作者 / 不是父评论作者 / 不是评论里提到的其他人
- 上下文只用于理解评论在讨论什么, 不要把上下文里的场景 / 设备 / 身份当成本作者的
- 反例: 原帖讲烘焙 → 评论说"那个搅拌器还行" → 评论作者**不一定**是烘焙爱好者

【推断硬约束】
- 只使用评论字面出现 + 上下文明确支持的信息
- 删掉这个词评论还能成立吗? 能 → 视作推断, 写"未提及"
- 当评论是对他人提问 / 观点的回复时, "决策阶段"优先选"评价他人"

【输出格式硬要求】
- 输出仅一个连贯的中文段落, 80-150 字
- 严禁任何 markdown 符号: 不要 **粗体**, 不要 # 标题, 不要 - 列表
- 严禁附录"维度分析": 不要在段落后追加 "1. 使用场景:..." 这样的列表
- 严禁前后缀: 不要"以下是画像:" 之类
- 6 个维度要自然融进段落叙述, 不要逐条列举
</rules>
